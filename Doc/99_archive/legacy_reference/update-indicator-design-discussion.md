# 「記録の更新可能性」インジケーター設計議論

## 前提条件の整理

### ユーザー要件
1. **同じ場所もしくはクラスタ単位で複数の登録がある事はよしとする**
   - 同じ場所に複数の記録があっても問題ない
   - クラスタ内で複数の記録を承認できる

2. **承認と更新の定義を明確にする**
   - **承認**: 記録が承認されること（複数の承認を許可）
   - **更新**: クラスタ内で「代表記録」を1つ選ぶこと

3. **複数の承認は許可するが、更新した情報はそのクラスタ内では１つのみにする**
   - クラスタ内で複数の記録を承認できる
   - ただし、クラスタ内で「代表記録」は1つのみ

## 必要な条件の整理

### 1. クラスタの定義

#### 現在の実装
- Mapboxのクラスタリング機能を使用
- 半径: `SPOT_MERGE_CONFIG.DISTANCE_THRESHOLD = 0.0005度 ≈ 50m`
- 同じ場所に複数の記録がある場合、`MultipleRecordsList`でリスト表示

**質問**: クラスタの半径は50mで問題ないでしょうか？それとも、更新可能性の判定には別の半径を使いますか？

### 2. 「代表記録」の選定基準

#### 提案: 代表記録の選定基準（優先順位順）

1. **難易度レベル**（最優先）
   - より高度な技をメイクした記録を優先
   - 例: Hardflip > Kickflip > Ollie

2. **承認者数**（2番目）
   - 承認者数が多い記録を優先
   - 例: 3人の承認者 > 2人の承認者 > 1人の承認者

3. **承認日時**（3番目）
   - より新しい承認日時の記録を優先
   - 例: 2024年 > 2023年

4. **記録日時**（4番目）
   - より新しい記録日時の記録を優先
   - 例: 2024年 > 2023年

#### 実装イメージ
```typescript

  // 2. 承認者数でソート
  // 3. 承認日時でソート
  // 4. 記録日時でソート

}
```

**質問**: この選定基準で問題ないでしょうか？他に考慮すべき基準はありますか？

### 3. 「更新」の定義

#### 提案: 更新の定義

**更新とは**: クラスタ内で「代表記録」を選定すること

- **自動更新**: 新しい記録が承認された時点で、クラスタ内の代表記録を自動的に再選定
- **手動更新**: 管理者が手動で代表記録を変更できる（将来的には）

#### 更新のタイミング
1. **新しい記録が承認された時**
   - クラスタ内の代表記録を自動的に再選定
   - より高度な技で記録が追加された場合、代表記録が更新される

2. **承認者数が増えた時**
   - クラスタ内の代表記録を自動的に再選定
   - 承認者数が多い記録が代表記録になる

**質問**: この更新の定義で問題ないでしょうか？

---

### 4. 「更新可能性」の判定基準

#### 提案: 更新可能性の判定

**更新可能な条件**:
1. クラスタ内に既存の記録がある
2. 新しい記録が、既存の代表記録より「高度」である可能性がある
   - より高度な技をメイクした場合
   - より多くの承認者を得た場合
   - より新しい記録の場合

#### 実装イメージ
```typescript
function canUpdateRecord(newRecord: Spot, representativeRecord: Spot): boolean {
  // 1. 難易度レベルを比較
  if (getDifficultyLevel(newRecord.trick) > getDifficultyLevel(representativeRecord.trick)) {
    return true
  }
  // 2. 承認者数を比較（将来的には）
  // 3. 記録日時を比較
  return false
}
```

**質問**: この判定基準で問題ないでしょうか？

### 6. 更新プロセス

#### 提案: 更新プロセスの流れ

1. **ユーザーが新しい記録を投稿**
   - 既存のクラスタ内の場所に記録を投稿
   - フォームに「この場所には既存の記録があります。より高度な技で更新できます」というメッセージを表示

2. **記録が承認される**
   - 承認プロセスは通常通り
   - 承認後、クラスタ内の代表記録を自動的に再選定

3. **代表記録が更新される**
   - より高度な技で記録が追加された場合、代表記録が自動的に更新される
   - 通知（将来的には））: 「この場所の代表記録が更新されました」

**質問**: この更新プロセスで問題ないでしょうか？

## 改善案

### 改善案1: クラスタの半径を調整可能にする

**提案**: 更新可能性の判定には、クラスタの半径とは別の半径を使う

```typescript
const UPDATE_POTENTIAL_CONFIG = {
  CLUSTER_RADIUS_METERS: 50, // クラスタの半径
  UPDATE_CHECK_RADIUS_METERS: 25, // 更新可能性の判定半径（より狭く）
}
```

**理由**: 更新可能性の判定をより厳密にするため

---

### 改善案2: 「更新可能性」の判定をより柔軟にする

**提案**: 難易度レベルだけでなく、記録の「質」も考慮する

```typescript
function canUpdateRecord(newRecord: Spot, representativeRecord: Spot): boolean {
  // 1. 難易度レベルを比較
  const newLevel = getDifficultyLevel(newRecord.trick)
  const currentLevel = getDifficultyLevel(representativeRecord.trick)
  
  if (newLevel > currentLevel) {
    return true
  }
  
  // 2. 同じ難易度レベルでも、より新しい記録を優先（将来的には）
  if (newLevel === currentLevel && isNewer(newRecord, representativeRecord)) {
    return true
  }
  
  return false
}
```

**理由**: より柔軟な更新判定のため
