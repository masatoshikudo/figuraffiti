# スポット申請・承認フロー

> **Phase1 注**: `/mapping` ではスポット申請フォームは非表示。`/submit` は存在するが、Phase1 設計では運営設置のみ想定。発見記録（NFC/QR タギング）は `/discover` で実行。

## フロー概要

1. **サインアップ・ログイン**
   - ユーザーがサインアップまたはログイン
   - 認証が必要（記録送信には認証必須）

2. **スポット情報を入力**（`/submit` ページ）
   - 地図で位置を選択
   - **場所名・文脈**: 入力
   - **メディアURL**: Instagram、YouTube 等（任意）

3. **記録を送信**
   - 「記録を残す」ボタンを押下
   - データベースに保存される

## 送信後の挙動

### 信頼ユーザーでない場合（一般ユーザー）

1. **ステータス**: `pending`（承認待ち）で保存される
2. **レスポンス**: 「投稿を受け付けました。承認後に公開されます。」というメッセージが返される
3. **表示**: 地図には表示されない（承認待ちのため）
4. **通知**:
   - ✅ **オーナー（管理者）へのメール通知**: 新しい投稿が承認待ちであることを通知
   - ✅ **ユーザーへのメール通知**: 承認/却下時に通知（承認時または却下時）

### 信頼ユーザーの場合

> **Phase1**: trusted_users テーブル廃止のため、全てのユーザーは承認待ち（pending）となる。

1. **ステータス**: `approved`（承認済み）で保存される（Phase1 では発生しない）
2. **レスポンス**: 作成されたスポットが返される
3. **表示**: すぐに地図に表示される（自動承認のため）
4. **通知**: なし（自動承認のため）

## 承認プロセス

1. **管理者が `/admin` ページにアクセス**
2. **承認待ちリストを確認**
3. **各スポットに対して承認/却下を実行**
   - **承認**: ステータスを `approved` に変更 → 地図に表示される
   - **却下**: ステータスを `rejected` に変更 → 地図に表示されない

## メール通知のタイミング

### 投稿時（新規投稿が承認待ちの場合）

- **送信先**: 管理者（`admin_users` テーブルに登録されているユーザー）
- **内容**: 
  - 件名: 「新しい記録が承認待ちです」
  - 本文: スポット名、投稿者、投稿日時、承認ページへのリンク

### 承認時

- **送信先**: 投稿者（`submitted_by` が設定されている場合）
- **内容**: 
  - 件名: 「スポットが承認されました」
  - 本文: スポット名、承認日時

### 却下時

- **送信先**: 投稿者（`submitted_by` が設定されている場合）
- **内容**: 
  - 件名: 「スポットが却下されました」
  - 本文: スポット名、却下理由、却下日時

## テストフロー

1. **サインアップ**: 新しいアカウントを作成
2. **ログイン**: 作成したアカウントでログイン
3. **`/submit` を開く**: 地図で位置を選択し、記録情報を入力
   - 場所名: 「渋谷スクランブル」
   - メディアURL: 「https://www.instagram.com/p/example」（任意）
4. **記録を送信**: 「記録を残す」ボタンを押下
5. **確認**:
   - フロントエンド: 「投稿を受け付けました。承認後に公開されます。」というメッセージが表示される
   - メール: 管理者にメール通知が送信される（実装予定）
   - データベース: `status='pending'` で保存される
6. **管理者が承認**:
   - `/admin` ページで承認待ちリストを確認
   - 承認ボタンを押下
   - ユーザーにメール通知が送信される
   - 地図に表示される
