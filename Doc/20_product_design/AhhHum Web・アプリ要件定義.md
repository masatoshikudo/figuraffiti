# AhhHum Web・アプリ 要件定義

本ドキュメントは、`AhhHum Webアプリ全体設計マスタードキュメント.md` を親設計として、その内容を**実装可能な機能要件・非機能要件**へ落とし込んだ要件定義である。  
親設計と本書に差分がある場合は、**親設計を優先**する。

## 1. 全体像

| レイヤー | Web/アプリが担う役割 | ユーザー価値 |
|---------|----------------------|-------------|
| **Discovery Layer** | SNS流入後の期待形成、世界観理解、重点区の入口形成 | 「何かが起きている」と感じる |
| **Exploration Layer** | 曖昧サークル、Last Seen、ヒント、重点区マップ | 「行ってみたい」「見つけたい」と感じる |
| **Proof Layer** | NFC/QR読み取り、発見順、FTF、デジタルタグ | 「自分が見つけた」と証明できる |
| **Expansion Layer** | コレクション、共鳴、Legend、物理キー、共創 | 「次も関わりたい」「自分も世界を増やしたい」と感じる |

このほか、**管理機能**（スポット管理、発見管理、共創審査、コンテンツ管理）と、Phase2以降の**EC / 物理キー導線**が必要になる。

---

## 2. フェーズ別スコープ

| フェーズ | 目的 | Web/アプリのスコープ |
|----------|------|----------------------|
| **Phase1** | 核の成立 | 曖昧サークル、Last Seen、ティッカー、NFC/QR発見記録、最小管理機能 |
| **Phase2** | 継続と収益 | コレクション、共鳴、Legend、物理キー、Shop |
| **Phase3** | 共創の成立 | 設置申請、審査、アンバサダー、設置者属性表示 |
| **Phase4** | IP基盤化 | 多都市圏展開、多言語、コラボ、外部ブランド接続 |

## 3. Phase1 必須要件

### 3.1 Identity Module

| ID | 要件 | 詳細 |
|----|------|------|
| I-1 | **認証** | 発見記録、デジタルタグ、プロフィール閲覧には認証を必須とする。マップ閲覧は未認証でも可。 |
| I-2 | **認証方式** | Phase1 は Supabase Auth を利用し、メール/パスワードを基本とする。 |
| I-3 | **プロフィール最小構成** | 表示名を保持し、ティッカーや発見ログに利用できるようにする。 |
| I-4 | **認証ゲート** | 発見記録など要認証行動の直前でモーダルまたは画面遷移によりログインを促す。 |

### 3.2 Discovery Map Module

| ID | 要件 | 詳細 |
|----|------|------|
| M-1 | **曖昧サークル表示** | スポットは正確なピンではなく、半径50m前後のサークルで表示する。 |
| M-2 | **鮮度表示** | `last_seen` をもとに、サークル色と相対時刻で鮮度を表現する。 |
| M-3 | **重点区中心の初期表示** | 都内23区のうち、重点区を中心にした初期表示とする。 |
| M-4 | **現在地補助** | 現在地周辺の探索を補助するUIを持つ。Phase1では簡易、Phase2以降で強化可。 |
| M-5 | **ティッカー表示** | 新規発見ログを1行ティッカーで表示し、世界が生きている感を作る。 |

### 3.3 Spot Detail Module

| ID | 要件 | 詳細 |
|----|------|------|
| S-1 | **サークル詳細表示** | サークルタップで Last Seen、ヒント、設置文脈、発見CTA を表示する。 |
| S-2 | **正確な位置は非表示** | 住所やピン位置は出さず、探索性を保つ。 |
| S-3 | **ヒント設計** | テキスト、シルエット、周辺テクスチャ、キャラクター視点写真などを表示できる構造を持つ。 |
| S-4 | **キャラクターページ連携** | スポット詳細から `characters/[slug]` へ遷移できる。 |

### 3.4 Proof Module

| ID | 要件 | 詳細 |
|----|------|------|
| P-1 | **NFC/QR 読み取り** | `discover/nfc` から読み取りまたは手動入力でスポットIDを受け取れる。 |
| P-2 | **発見記録作成** | 読み取り成功時に discovery レコードを作成し、`last_seen` を更新する。 |
| P-3 | **発見順の確定** | 初回発見者か、何番目の発見者かを返せる構造を持つ。 |
| P-4 | **成功演出** | FTF、発見順、Just now 更新を表示する。 |
| P-5 | **デジタルタグ** | 発見者が短いタグや一言を残せるようにする。Phase1は最小、Phase2で強化。 |

### 3.5 Admin Core Module

| ID | 要件 | 詳細 |
|----|------|------|
| A-1 | **スポット管理** | スポットの作成、編集、公開、休眠管理ができる。 |
| A-2 | **発見管理** | 発見ログ一覧、異常検知、重複確認、必要な修正ができる。 |
| A-3 | **管理者ロール** | 管理画面は管理者のみアクセス可能とする。 |
| A-4 | **QR/NFC URL管理** | スポットまたはキャラクターに紐づく読み取りURLを確認できる。 |

## 4. Phase2 追加要件

| ID | 要件 | 詳細 |
|----|------|------|
| X-1 | **コレクション** | 発見済み / 未発見 / 共鳴条件を一覧できる。 |
| X-2 | **共鳴 / シークレット** | 特定条件達成で新サークルやシークレットスポットを解放する。 |
| X-3 | **Legends** | 休眠スポットを金色サークルで閲覧できる。 |
| X-4 | **物理キー連携** | 物理キー保有者だけが再覚醒に必要な正確位置へ進める。 |
| X-5 | **Shop** | 物理キー、観賞用フィギュア、限定商品を販売できる。 |
| X-6 | **SNS埋め込み強化** | Instagram / TikTok 等の制作過程や発見動画をサイト内で表示できる。 |

## 5. Phase3 追加要件

| ID | 要件 | 詳細 |
|----|------|------|
| C-1 | **設置申請** | ユーザーが設置意図、場所、写真を添えて申請できる。 |
| C-2 | **共創審査** | ブランド基準に基づいて承認 / 却下 / 差し戻しができる。 |
| C-3 | **設置者属性表示** | 発見時に運営 / 一般 / アンバサダーを表示できる。 |
| C-4 | **アンバサダー制度** | 招待制で上位設置者を管理できる。 |

## 6. 非機能要件

| ID | 要件 | 詳細 |
|----|------|------|
| NF-1 | **対応デバイス** | 第一に**スマートフォン**（縦画面）。タブレット・PCでも利用可能とする。 |
| NF-2 | **対応ブラウザ** | 主要モバイルブラウザ（iOS Safari, Android Chrome）およびデスクトップChrome, Safari, Edge。 |
| NF-3 | **パフォーマンス** | 地図の初期表示は3秒以内を目安とする。読み取り後の成功反映は即時感を持たせる。 |
| NF-4 | **可用性** | 想定トラフィックに応じたホスティング（Vercel / Supabase等）とし、ダウンタイムを最小化する。 |
| NF-5 | **セキュリティ** | 認証はSupabase Auth等の実績ある方式を用いる。管理画面はHTTPS・認可の二重チェック。個人情報は適切に保護する。 |
| NF-6 | **運用・保守** | コンテンツ（キャラ情報・スポット・ヒント）の更新を、コードを触らずにできるようにする。 |
| NF-7 | **スマホ最適** | 屋外利用前提で、縦画面・片手操作・弱回線に最適化する。 |
| NF-8 | **探索性維持** | 正確な位置を出しすぎず、発見の快感を損なわない。 |

## 9. まとめ: Phase1で実装すべき最小セット

1. **オンラインマップ**  
   曖昧サークル、Last Seen、ティッカーを表示する。

2. **サークル詳細**  
   ヒントと文脈、発見CTAを提示する。

3. **発見記録**  
   NFC/QR読み取り、発見順判定、成功演出、鮮度更新を行う。

4. **認証・プロフィール**  
   発見記録に必要な認証と、最小プロフィールを提供する。

5. **管理機能**  
   スポット、発見、管理者ロールを扱えるようにする。

6. **親設計との整合**  
   画面ルート、画面ID、Phase構成は `AhhHum Webアプリ全体設計マスタードキュメント.md` に従う。

Phase2では、コレクション、共鳴、Legend、物理キー、Shop を追加する。  
Phase3では、設置申請と共創審査を追加する。
