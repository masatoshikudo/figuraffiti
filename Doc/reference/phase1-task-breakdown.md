# Phase 1 実装タスク分解

## Phase 1の目標
- 「記録の更新可能性」インジケーターの実装
- 「承認済み記録の集中エリア」ヒートマップの実装

## タスク分解

### Task 1-1: 更新可能性判定ロジックの拡張とテスト
**目的**: 既存の`canUpdateRecord()`関数を拡張し、地図上の任意の位置で更新可能性を判定できるようにする

**実装内容**:
- [ ] `lib/spot/spot-utils.ts`に`findUpdatableSpots()`関数を追加
  - 指定位置から一定距離内の承認済み記録を取得
  - 各記録について、より高難易度の技で更新可能かを判定
  - 更新可能な記録のリストを返す
- [ ] 更新可能性判定の閾値（距離）を定数として定義（`lib/constants.ts`）
- [ ] ユニットテストまたは動作確認用のダミーデータでテスト

**依存関係**: なし（既存の`getDifficultyLevel()`と`canUpdateRecord()`を使用）

**推定工数**: 2-3時間

---

### Task 1-2: 更新可能性インジケーターのUIコンポーネント作成
**目的**: 更新可能な記録に表示するバッジ/インジケーターのUIコンポーネントを作成

**実装内容**:
- [ ] `components/map/update-indicator.tsx`を作成
  - 更新可能性バッジの表示（🔄アイコン + テキスト）
  - ホバー時のツールチップ表示（現在の記録情報）
  - クリック時の動作（更新フォームを開く）
- [ ] スタイリング（Tailwind CSS）
  - バッジの色、サイズ、アニメーション
  - ホバーエフェクト

**依存関係**: Task 1-1

**推定工数**: 3-4時間

---

### Task 1-3: 地図上への更新可能性インジケーターの統合
**目的**: MapViewコンポーネントに更新可能性インジケーターを統合し、地図上に表示

**実装内容**:
- [ ] `components/map/map-view.tsx`を更新
  - 承認済み記録に対して更新可能性を判定
  - 更新可能な記録の位置にインジケーターを表示
  - マーカーとインジケーターの重複を避ける配置ロジック
- [ ] パフォーマンス最適化
  - 表示範囲内の記録のみを判定（ビューポート計算）
  - デバウンス処理（ズーム・パン時の再計算を抑制）

**依存関係**: Task 1-1, Task 1-2

**推定工数**: 4-5時間

---

### Task 1-4: 更新フォームへの統合（更新モード対応）
**目的**: 更新可能性インジケーターから更新フォームを開き、既存記録を参照できるようにする

**実装内容**:
- [ ] `components/form/submit-form-bottom-sheet.tsx`を更新
  - 更新モードの追加（新規投稿モードと更新モード）
  - 更新モード時は既存記録の情報を表示
  - フォーム送信時に既存記録IDを参照
- [ ] 更新フォームのUI調整
  - 「この記録を更新します」というメッセージ表示
  - 現在の記録情報（技名、難易度レベル）を表示

**依存関係**: Task 1-3

**推定工数**: 3-4時間

---

### Task 1-5: ヒートマップ用データ準備関数の実装
**目的**: 承認済み記録からヒートマップ用のデータを生成する関数を作成

**実装内容**:
- [ ] `lib/spot/spot-utils.ts`に`generateHeatmapData()`関数を追加
  - 承認済み記録のみをフィルタリング
  - Mapboxヒートマップレイヤー用のGeoJSON形式に変換
  - 座標配列を返す（`[lng, lat, intensity]`形式）
- [ ] ヒートマップの強度計算ロジック
  - 記録の集中度に応じた強度の計算（オプション）

**依存関係**: なし

**推定工数**: 2-3時間

---

### Task 1-6: Mapboxヒートマップレイヤーの実装
**目的**: Mapbox GL JSのヒートマップレイヤーを地図に追加

**実装内容**:
- [ ] `components/map/map-view.tsx`を更新
  - Mapboxの`addSource()`と`addLayer()`を使用してヒートマップレイヤーを追加
  - ヒートマップのスタイル設定（色、半径、強度）
  - レイヤーの表示/非表示を切り替える機能（オプション）
- [ ] ヒートマップと既存マーカーの重複表示の調整
  - ヒートマップがマーカーを隠さないように調整

**依存関係**: Task 1-5

**推定工数**: 4-5時間

---

### Task 1-7: ヒートマップのスタイリングとインタラクション
**目的**: ヒートマップの見た目を調整し、ユーザーが操作できるようにする

**実装内容**:
- [ ] ヒートマップの色設定
  - 承認済み記録の集中度に応じた色のグラデーション
  - 低密度: 青系、中密度: 緑系、高密度: 赤系
- [ ] ヒートマップの透明度調整
  - マーカーとの視認性を考慮した透明度設定
- [ ] ヒートマップの表示切り替え機能（オプション）
  - トグルボタンでヒートマップの表示/非表示を切り替え

**依存関係**: Task 1-6

**推定工数**: 2-3時間

---

### Task 1-8: 統合テストとパフォーマンス最適化
**目的**: Phase 1の全機能を統合し、動作確認とパフォーマンス最適化を行う

**実装内容**:
- [ ] 統合テスト
  - 更新可能性インジケーターとヒートマップが同時に表示される状態でテスト
  - 大量の記録データでのパフォーマンステスト
- [ ] パフォーマンス最適化
  - 不要な再計算の削減
  - メモ化の適用
  - レンダリング最適化
- [ ] エラーハンドリング
  - ヒートマップデータが空の場合の処理
  - 更新可能性判定エラーの処理

**依存関係**: Task 1-3, Task 1-6, Task 1-7

**推定工数**: 3-4時間

---

## 実装順序の推奨

1. **Task 1-1** → **Task 1-2** → **Task 1-3** → **Task 1-4**（更新可能性インジケーター）
2. **Task 1-5** → **Task 1-6** → **Task 1-7**（ヒートマップ）
3. **Task 1-8**（統合テスト）

## 合計推定工数
- 最小: 23時間
- 最大: 31時間

## 注意事項

1. **データベーススキーマ**: 現時点では追加のマイグレーションは不要（既存の`status`カラムで承認済みを判定可能）
2. **パフォーマンス**: 記録数が増えた場合のパフォーマンスを考慮し、必要に応じてバックエンドAPIで集計処理を追加する可能性がある
3. **UI/UX**: ヒートマップとマーカーの視認性を考慮し、必要に応じて表示切り替え機能を追加
