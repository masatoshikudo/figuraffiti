# `/mapping`ページにピン打ち機能を追加する設計

## UX的な利点

### 方法1（地図をクリックしてピンを打つ）の利点

1. **直感的な操作**
   - 地図を見ながら「ここに記録したい」と思った瞬間にクリックできる
   - 場所を選んでから記録するという自然な流れ

2. **コンテキストの維持**
   - 地図を見ながら記録できるため、周辺の記録と比較しやすい
   - 「この場所にはまだ記録がない」という発見がしやすい

3. **操作の簡略化**
   - `/submit`ページに移動する必要がない
   - ワンクリックで記録開始

4. **即座のフィードバック**
   - 記録を投稿すると、その場で地図に反映される
   - 記録が承認されると、同じ場所にピンが表示される

## 実装設計

### 機能要件

1. **地図クリックでピン表示**
   - 地図をクリックすると、一時的なピン（仮ピン）が表示される
   - 仮ピンは既存のスポットピンと視覚的に区別できる（例: グレー、点線、アニメーション）

2. **ピンクリックで記録フォームを開く**
   - 仮ピンをクリックすると、記録フォーム（ボトムシート）が開く
   - フォームには、クリックした位置の座標が自動入力される

3. **記録フォームの内容**
   - 場所: 自動入力（クリックした位置）
   - 技名: 入力フィールド
   - メディアURL: 入力フィールド
   - 送信ボタン

4. **記録投稿後の処理**
   - 記録を投稿すると、仮ピンは削除される
   - 記録は「承認待ち」状態で保存される
   - 承認されると、地図に正式なピンとして表示される

### UI/UXの詳細

#### 仮ピンのデザイン

- **色**: グレー（`#9CA3AF`）または半透明
- **スタイル**: 点線のボーダー、またはアニメーション（パルス効果）
- **サイズ**: 既存のピンより少し大きめ（目立たせるため）
- **アイコン**: プラス記号（`+`）またはピンアイコン

#### 記録フォーム（ボトムシート）

- **開き方**: 仮ピンをクリック、または地図をクリックした直後に自動で開く
- **内容**: 
  - 場所の確認（座標と逆ジオコーディングした住所）
  - 技名入力
  - メディアURL入力
  - 送信ボタン
  - キャンセルボタン（仮ピンを削除）

#### 操作フロー

1. ユーザーが地図をクリック
2. クリック位置に仮ピンが表示される
3. 仮ピンをクリック（または自動で）記録フォームが開く
4. ユーザーが記録情報を入力
5. 送信ボタンをクリック
6. 記録が投稿される（承認待ち）
7. 仮ピンが削除される
8. 成功メッセージが表示される

   ```typescript
   }
   ```

   ```typescript
   const handleSubmit = async (data: { trickName: string; mediaUrl: string }) => {
   ```

### 既存機能との統合

#### `/submit`ページとの関係

- `/submit`ページは引き続き存在する（明示的に記録したい場合）
- `/mapping`ページから記録できるようになる（発見的な記録）
- 両方の方法で記録できるため、ユーザーの選択肢が増える

#### 承認フローの維持

- 記録は引き続き承認制
- 承認待ちの記録は地図に表示されない
- 承認されると、地図に正式なピンとして表示される

### 実装の優先順位

1. **Phase 1: 基本機能**
   - 地図クリックで仮ピン表示
   - 仮ピンクリックで記録フォームを開く
   - 記録フォームの表示

2. **Phase 2: フォーム機能**
   - 場所の確認表示
   - 技名入力
   - メディアURL入力
   - 送信機能

3. **Phase 3: UX改善**
   - 仮ピンのアニメーション
   - 送信後のフィードバック
   - エラーハンドリング

### 考慮事項

1. **既存のスポットピンとの競合**
   - 既存のスポットピンをクリックした場合は、記録フォームではなく詳細を表示
   - 地図の空白部分をクリックした場合のみ仮ピンを表示

2. **モバイル対応**
   - タッチイベントにも対応
   - ボトムシートはモバイルで使いやすい

3. **パフォーマンス**
   - 仮ピンは1つだけ表示（新しい位置をクリックすると移動）
   - 記録フォームは必要時のみ表示

## SP版（スマートフォン版）のUXフロー

### SP版特有の考慮事項

1. **タッチ操作**
   - クリックではなくタップ
   - 誤タップの防止が必要
   - スクロールとタップの競合を避ける

2. **画面サイズ**
   - 画面が小さいため、ボトムシートが適切
   - 仮ピンが画面の大部分を占めないようにする

3. **操作性**
   - 片手操作を考慮
   - 重要なボタンは親指が届く範囲に配置

### SP版の操作フロー

#### フロー1: 地図をタップしてピンを打つ（基本フロー）

1. **地図をタップ**
   - ユーザーが地図の空白部分をタップ
   - 既存のスポットピンではないことを確認

2. **仮ピンが表示される**
   - タップ位置に仮ピンが表示される
   - 仮ピンは視覚的に目立つ（アニメーション、グレー、点線）
   - 仮ピンは少し大きめ（タップしやすくするため）

3. **自動でボトムシートが開く（推奨）**
   - 仮ピンが表示されたら、自動で記録フォームのボトムシートが開く
   - ユーザーは追加のタップ不要

4. **記録フォームを入力**
   - ボトムシート内で記録情報を入力
   - 場所は自動入力（タップした位置）
   - 技名、メディアURLを入力

5. **送信**
   - 送信ボタンをタップ
   - 記録が投稿される

6. **フィードバック**
   - ボトムシートが閉じる
   - 仮ピンが削除される
   - 成功メッセージが表示される

#### フロー2: 仮ピンをタップしてフォームを開く（代替フロー）

1. **地図をタップ**
   - 仮ピンが表示される
   - ボトムシートは自動で開かない

2. **仮ピンをタップ**
   - 仮ピンをタップすると記録フォームが開く
   - ユーザーが明示的にフォームを開く操作

3. **以降はフロー1と同じ**

### SP版のUI/UX詳細

#### 仮ピンのデザイン（SP版）

- **サイズ**: 32px × 32px（PC版より大きめ）
- **色**: グレー（`#9CA3AF`）または半透明
- **スタイル**: 
  - 点線のボーダー（2px）
  - パルスアニメーション（1秒間隔）
  - 影を付けて目立たせる
- **アイコン**: プラス記号（`+`）またはピンアイコン
- **位置**: タップ位置の中央に配置

#### ボトムシートのデザイン（SP版）

- **高さ**: 画面の60-80%（初期は60%、キーボード表示時は80%）
- **ドラッグハンドル**: 上部にドラッグハンドルを表示
- **スワイプで閉じる**: 下にスワイプすると閉じる（キャンセル）
- **背景**: 半透明のオーバーレイ（地図が見える）

#### 記録フォームのレイアウト（SP版）

```
┌─────────────────────────┐
│  [ドラッグハンドル]      │
│  記録を残す              │
│                         │
│  📍 場所                 │
│  東京都渋谷区...         │
│                         │
│  技名 *                 │
│  [入力フィールド]        │
│                         │
│  メディアURL *          │
│  [入力フィールド]        │
│                         │
│  [キャンセル] [送信]     │
└─────────────────────────┘
```

#### 誤タップの防止

1. **デバウンス処理**
   - タップ後、200ms以内の連続タップを無視
   - スクロール中のタップを無視

2. **タップ領域の拡大**
   - 仮ピンのタップ領域を拡大（44px × 44px）
   - 既存のスポットピンとの競合を避ける

3. **確認ダイアログ（オプション）**
   - 初回タップ時のみ確認ダイアログを表示
   - 「この場所に記録を追加しますか？」

### SP版の操作パターン

#### パターン1: 素早い記録（推奨）

1. 地図をタップ
2. 自動でボトムシートが開く
3. 情報を入力
4. 送信

**所要時間**: 約30秒

#### パターン2: 慎重な記録

1. 地図をタップ
2. 仮ピンを確認
3. 仮ピンをタップ
4. ボトムシートが開く
5. 情報を入力
6. 送信

**所要時間**: 約1分

### SP版の考慮事項

#### 1. スクロールとタップの競合

- **問題**: 地図をスクロールしようとして誤ってタップしてしまう
- **解決策**: 
  - スクロール中（指が動いている間）はタップイベントを無視
  - タップと判定する閾値を設定（5px以内の移動）

#### 2. キーボード表示時のレイアウト

- **問題**: キーボードが表示されると、ボトムシートが隠れる
- **解決策**:
  - キーボード表示時にボトムシートの高さを調整
  - スクロール可能にする
  - 入力フィールドがキーボードに隠れないようにする

#### 3. 片手操作の考慮

- **問題**: 片手で操作しにくい
- **解決策**:
  - 送信ボタンを下部に配置（親指が届く範囲）
  - キャンセルボタンも下部に配置
  - ボタンサイズを大きくする（44px × 44px以上）

#### 4. 既存ピンとの競合

- **問題**: 既存のスポットピンと仮ピンが重なる
- **解決策**:
  - 既存のスポットピンをタップした場合は、記録フォームではなく詳細を表示
  - 地図の空白部分をタップした場合のみ仮ピンを表示
  - 既存ピンから一定距離（50px）以上離れた位置のみ仮ピンを表示

### SP版の実装優先順位

1. **Phase 1: 基本機能**
   - 地図タップで仮ピン表示
   - 自動でボトムシートを開く
   - 記録フォームの表示

2. **Phase 2: UX改善**
   - 誤タップの防止（デバウンス、スクロール検知）
   - キーボード表示時のレイアウト調整
   - アニメーションの追加

3. **Phase 3: 高度な機能**
   - スワイプで閉じる
   - ドラッグハンドル
   - 確認ダイアログ（オプション）

### SP版のテスト項目

1. **操作性**
   - 片手操作で使いやすいか
   - 誤タップが発生しないか
   - スクロールとタップが競合しないか

2. **視認性**
   - 仮ピンが見やすいか
   - ボトムシートが適切な高さか
   - キーボード表示時にレイアウトが崩れないか

3. **パフォーマンス**
   - アニメーションがスムーズか
   - タップの応答性が良いか
   - メモリリークがないか

## 結論

方法1（地図をクリックしてピンを打つ）は、UX的に優れており、記録主役の設計とも一致しています。実装することで、ユーザーは地図を見ながら自然に記録を残せるようになります。

SP版では、タッチ操作に最適化し、誤タップを防止し、片手操作を考慮した設計が重要です。特に、自動でボトムシートが開くフローは、SP版では操作を簡略化し、UXを向上させます。
