# 承認制システム設計書

## 概要
スパム・不正投稿対策として、投稿を承認制にし、事前に信用がある人からの投稿に限定するシステムを実装します。

## 基本方針

### 1. 承認制の導入
- **導入時期**: すぐに導入
- **ステータス**: `pending`（承認待ち）、`approved`（承認済み）、`rejected`（却下）
- **既存データ**: すべて`approved`として扱う

### 2. 投稿要件
- **認証**: 不要（匿名投稿も可能）
- **初期ステータス**: `pending`（承認待ち）
- **信頼ユーザー**: 自動承認（`approved`で投稿）

### 3. 信頼ユーザー管理
- **昇格方法**: 手動のみ（自動昇格なし）
- **管理**: 管理画面から手動で追加/削除
- **削除時の扱い**: 既存の自動承認済み投稿はそのまま承認済みのまま

### 4. 承認プロセス
- **承認方法**: 管理画面から個別承認
- **却下理由**: 必須入力
- **再提出**: 許可する（新しい投稿として扱う）

### 5. 通知システム
- **通知対象**: 認証済みユーザーのみ（匿名投稿は通知なし）
- **通知方法**: メールのみ
- **通知タイミング**: 
  - 承認時: スポット名、承認日時
  - 却下時: スポット名、却下理由、却下日時

## データベース設計

### spotsテーブルの追加カラム
```sql
- status: TEXT DEFAULT 'pending' CHECK (status IN ('pending', 'approved', 'rejected'))
- submitted_by: UUID REFERENCES auth.users(id) ON DELETE SET NULL (NULL許可: 匿名投稿対応)
- approved_by: UUID REFERENCES auth.users(id) ON DELETE SET NULL
- approved_at: TIMESTAMP WITH TIME ZONE
- rejection_reason: TEXT (却下理由、必須)
```

### trusted_usersテーブル
```sql
- id: SERIAL PRIMARY KEY
- user_id: UUID NOT NULL UNIQUE REFERENCES auth.users(id) ON DELETE CASCADE
- created_at: TIMESTAMP WITH TIME ZONE DEFAULT NOW()
- created_by: UUID REFERENCES auth.users(id) ON DELETE SET NULL
- note: TEXT
```

## RLSポリシー

### 公開表示
- **承認済みスポット**: すべてのユーザーが閲覧可能
- **承認待ち/却下**: 投稿者本人のみ閲覧可能（認証済みの場合）

### 投稿
- **一般ユーザー**: `status='pending'`で投稿可能（認証不要）
- **信頼ユーザー**: `status='approved'`で投稿可能（自動承認）

### 管理
- **管理者**: すべてのスポットを閲覧・更新・削除可能
- **承認/却下**: 管理者のみ実行可能

## API設計

### 既存エンドポイントの変更

#### `POST /api/spots`
- **変更点**: 
  - 認証不要（匿名投稿可）
  - `status='pending'`を自動設定
  - 認証済みユーザーの場合、`submitted_by`を設定
- **レスポンス**: 作成されたスポット（承認待ち）

#### `GET /api/spots`
- **変更点**: 
  - デフォルトで`status='approved'`のスポットのみ返す
  - 認証済みユーザーは自分の投稿（承認待ち/却下含む）も閲覧可能

### 新規エンドポイント

#### `POST /api/spots/approve`
- **認証**: 必須（管理者のみ）
- **パラメータ**: `spotId` (string)
- **処理**: 
  - スポットの`status`を`approved`に変更
  - `approved_by`と`approved_at`を設定
  - 認証済みユーザーの場合、メール通知を送信
- **レスポンス**: 承認されたスポット

#### `POST /api/spots/reject`
- **認証**: 必須（管理者のみ）
- **パラメータ**: 
  - `spotId` (string)
  - `rejectionReason` (string, 必須)
- **処理**: 
  - スポットの`status`を`rejected`に変更
  - `rejection_reason`を設定
  - 認証済みユーザーの場合、メール通知を送信
- **レスポンス**: 却下されたスポット

#### `GET /api/spots/pending`
- **認証**: 必須（管理者のみ）
- **処理**: 承認待ちのスポット一覧を返す
- **レスポンス**: スポット配列

#### `GET /api/spots/my-submissions`
- **認証**: 必須
- **処理**: 認証済みユーザーが投稿したスポット一覧を返す（全ステータス）
- **レスポンス**: スポット配列

#### `GET /api/admin/trusted-users`
- **認証**: 必須（管理者のみ）
- **処理**: 信頼ユーザー一覧を返す
- **レスポンス**: ユーザー配列

#### `POST /api/admin/trusted-users`
- **認証**: 必須（管理者のみ）
- **パラメータ**: 
  - `userId` (string, UUID)
  - `note` (string, 任意)
- **処理**: 信頼ユーザーを追加
- **レスポンス**: 追加されたユーザー

#### `DELETE /api/admin/trusted-users/:userId`
- **認証**: 必須（管理者のみ）
- **処理**: 信頼ユーザーを削除
- **レスポンス**: 成功メッセージ

## 管理画面の機能

### 承認管理タブ
- **承認待ちリスト**: 
  - フィルタ: ステータス、投稿者、日付
  - 各スポットに承認/却下ボタン
  - 却下時は理由入力ダイアログ
- **承認済み/却下済み履歴**: 過去の承認/却下履歴を表示

### 信頼ユーザー管理タブ
- **信頼ユーザー一覧**: 
  - ユーザーID、追加日時、追加者、備考を表示
  - 削除ボタン
- **信頼ユーザー追加フォーム**: 
  - ユーザーID入力
  - 備考入力（任意）

## 通知システム

### メール通知（Supabase FunctionsまたはNext.js API Route）

#### 承認通知
- **送信条件**: 認証済みユーザーが投稿したスポットが承認された時
- **内容**: 
  - 件名: 「スポットが承認されました」
  - 本文: スポット名、承認日時、スポットへのリンク

#### 却下通知
- **送信条件**: 認証済みユーザーが投稿したスポットが却下された時
- **内容**: 
  - 件名: 「スポットが却下されました」
  - 本文: スポット名、却下理由、却下日時

### 実装方法
- **案A**: Supabase Edge Functions（推奨）
- **案B**: Next.js API Route + メール送信サービス（SendGrid、Resend等）

## 実装順序

1. **データベースマイグレーション**
   - `spots`テーブルに承認関連カラムを追加
   - `trusted_users`テーブルを作成
   - RLSポリシーを更新
   - 既存データを`approved`に更新

2. **型定義の更新**
   - `Spot`型に承認関連フィールドを追加
   - `spot-converter.ts`を更新

3. **APIルートの更新**
   - `POST /api/spots`を更新（認証不要、status=pending）
   - `GET /api/spots`を更新（承認済みのみ表示）
   - 新規エンドポイントを実装

4. **通知システムの実装**
   - メール送信機能を実装
   - 承認/却下時に通知を送信

5. **管理画面の更新**
   - 承認管理タブを追加
   - 信頼ユーザー管理タブを追加

6. **フロントエンドの更新**
   - 投稿フォームに承認待ちメッセージを表示
   - 自分の投稿一覧ページを追加（オプション）

## セキュリティ考慮事項

1. **管理者権限の確認**: すべての管理APIで管理者権限をチェック
2. **RLSポリシー**: データベースレベルでアクセス制御
3. **入力検証**: 却下理由の必須チェック、XSS対策
4. **レート制限**: 将来的に追加予定（今回は実装しない）

## 今後の拡張可能性

1. **自動承認ルール**: 一定条件を満たす投稿を自動承認
2. **アプリ内通知**: メール通知に加えてアプリ内通知
3. **承認履歴の詳細表示**: 誰がいつ承認/却下したかの履歴
4. **バッチ承認**: 複数のスポットを一括承認
5. **コンテンツフィルタリング**: 不適切な投稿の自動検出
